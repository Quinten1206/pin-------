# DeskPins 项目复盘总结

## 项目目标

制作一个 Windows 窗口置顶工具（仿 DeskPins）：常驻系统托盘、点击进入图钉模式、点击窗口置顶、DLL 注入绘制标题栏图标。

## 技术选型问题

### 选择了 C++ / MinGW-w64 / CMake，这是最差的选择之一

| 方案 | 优缺点 |
|------|--------|
| **C# / .NET + WinForms** | 最佳选择。托盘图标、光标、资源嵌入全是框架自带，DLL 注入用 P/Invoke 调同样的 Win32 API。`dotnet publish --self-contained` 输出单文件 exe，真正零依赖。代码量只有 C++ 的 1/3。 |
| **C++ / MSVC + Visual Studio** | 次优选择。Windows SDK 原生支持，调试器强大，资源编辑器可用，编译出的二进制是真正的 Windows 原生程序。 |
| **C++ / MinGW-w64 + CMake（实际使用）** | 最差选择。g++ 来自 GNU/Linux 生态，MSYS2 只是移植层。在 Windows 上永远是"二等公民"。 |

Mac/Windows 兼容性上，Windows 是非主流的那方，用 Linux 的工具链做 Win32 GUI 项目等于给自己全程添堵。

### 为什么会选 MinGW

VS Code 已经配好了 `D:\msys64\ucrt64\bin\g++.exe`（MSYS2 提供的 g++），所以就顺着用了。但这意味着**每一步编译、链接、资源处理都依赖 MSYS2 环境**，每个环节都会踩坑。

## 构建过程中遇到的问题

### 1. cmake 未安装

MinGW 不自带 cmake，需要单独通过 MSYS2 的 pacman 安装：

```
pacman -S mingw-w64-ucrt-x86_64-cmake
```

### 2. cmake 找不到可用的编译器

当前 shell（Git Bash / VS Code 终端）不在 MSYS2 环境内，PATH 里没有 `D:\msys64\ucrt64\bin`。解决办法是在运行 cmake 前导出 PATH：

```bash
export PATH="/d/msys64/ucrt64/bin:$PATH"
```

### 3. g++ 编译静默失败（cc1plus.exe 无法加载）

g++ 的主进程启动正常（`g++ --version` 能跑），但编译器核心 `cc1plus.exe` 加载时找不到同目录下的运行时 DLL（`libstdc++-6.dll`、`libgcc_s_seh-1.dll` 等），直接退出并返回错误码 1，没有任何报错信息。

同样是因为 shell 的 PATH 里没有 MSYS2 的 bin 目录。

### 4. `GET_X_LPARAM` 宏未定义

MSVC 的 `<windows.h>` 自动包含 `<windowsx.h>` 里的这些辅助宏，但 MinGW 不会。需要手动 `#include <windowsx.h>` 或用 `LOWORD`/`HIWORD` 手动拆解。

### 5. `GetDpiForWindow` 函数未声明

这是 Windows 10 1607 引入的 API。MinGW 默认的 `WINVER` 值较低，不会声明它。需要通过 CMake 全局定义：

```cmake
add_definitions(-DWINVER=0x0A00 -D_WIN32_WINNT=0x0A00)
```

### 6. `RT_RCDATA` 类型不兼容

`RT_RCDATA` 在 MinGW 里定义为 ANSI 版本（`MAKEINTRESOURCE(10)` → `LPSTR`），不能直接传给 Unicode API（`FindResourceW` 需要 `LPCWSTR`），需要显式强制转换 `(LPCWSTR)RT_RCDATA`。MSVC 的 Unicode 头文件有对应的宽字符宏，不会有这个问题。

### 7. 风资源编译器（windres）找不到 DLL 文件

`.rc` 文件在 `src/` 目录，windres 的工作目录也在 `src/`，而 DLL 在 `build/dll/`。需要在编译资源前通过 CMake 的 `PRE_BUILD` 自定义命令把 DLL 复制到 `src/` 目录下。

### 8. 空资源文件导致 windres 报错

DLL 的 `resource.rc` 最初是空的（只有 `#include "resource.h"`），windres 遇到空资源文件直接报错 "no resources"。解决办法：从 DLL 的 CMakeLists.txt 中暂时移除 `resource.rc`。

### 9. libstdc++-6.dll 动态依赖

默认 MinGW 编译链接是动态的，生成的 EXE 依赖 `libstdc++-6.dll`。发给别人无法运行。解决办法：加 `-static` 链接选项，把 C++ 标准库编译进 EXE。

## 流程问题

### Subagent-Driven Development 在这个项目上完全失败

分派了 3 个"实现子代理"，几乎每个都在环境问题上卡住（cmake 找不到、g++ 静默失败、windres 报错），每个都要多次来回沟通。最后干脆自己全写了，多绕了几圈冤枉路。对于环境敏感的本地项目，分派子代理只会放大问题，不如直接动手。

### 程序未验证

虽然编译通过了，但实际行为（托盘菜单、鼠标钩子、DLL 注入、标题栏绘制）没有在真实 Windows 桌面上测试过。

## 正确做法

如果重来，直接选 **C# + .NET 8 + WinForms**：

- `NotifyIcon` 控件：一行代码搞定托盘图标和右键菜单
- `Cursor` 类：换光标是 `this.Cursor = Cursors.Cross` 这种级别
- 资源嵌入：拖进项目就行，编译自动嵌入
- DLL 注入：P/Invoke 调用同样的 `SetWindowsHookEx`、`CreateRemoteThread`、`SetWindowSubclass`
- 发布：`dotnet publish -p:PublishSingleFile=true --self-contained` 输出单个 exe
- 代码量：全项目 200-300 行，不是现在的 20+ 个文件、600+ 行 C++

## 编译产物的局限

`DeskPins.exe`（930KB）和 `DeskPinsHook.dll`（104KB）目前已编译成功并存于 `build/` 目录下，但存在以下未验证的问题：

1. 实际运行行为未经测试（托盘图标是否出现、鼠标钩子是否工作、DLL 注入是否成功）
2. 标题栏图标绘制的坐标计算可能在 Windows 11 DWM 下偏移
3. 占位图标（assets/app.ico、assets/pin.cur）只是最小有效文件，视觉效果差
